Personal AI Memory.
План реализации. Подробно.

PHASE 01

AI-history hub MVP

~1-2 недели

Все твои разговоры с ChatGPT / Claude / Gemini автоматически собираются в локальную базу. Есть веб-интерфейс с базовым текстовым поиском. Никаких внешних API ещё не нужно.

// Что нового в этой фазе

Plasmo extension FastAPI backend Postgres схема Базовый Next.js UI

// Шаги

1

Setup инфраструктуры

~½ дня

Подготовка локальной среды. Никакого кода ещё, только tooling.

  • Установить Docker Desktop, Node.js 20+, Python 3.11+
  • Создать структуру: pam/backend, pam/extension, pam/web
  • Поднять Postgres + pgvector через docker-compose.yml
  • Python venv в backend/, установить FastAPI и uvicorn
# docker-compose.yml — минимальный конфиг
services:
  db:
    image: pgvector/pgvector:pg16
    ports: ["5432:5432"]
    environment:
      POSTGRES_PASSWORD: pam
      POSTGRES_DB: pam
    volumes: ["./data:/var/lib/postgresql/data"]
2

Backend skeleton

~1 день

FastAPI приложение с базовыми эндпоинтами. Минимум — UPSERT разговора и его получение.

  • SQLAlchemy модели: Conversation, Message
  • Миграции через Alembic
  • CORS-middleware для разрешения запросов от chrome-extension://*
  • Endpoints: POST /conversations (UPSERT), GET /conversations, GET /conversations/{id}, GET /search?q=
  • Поиск через PostgreSQL tsvector + websearch_to_tsquery (быстрее чем ILIKE и поддерживает русский)
# Минимальная схема таблиц
CREATE TABLE conversations (
  id          UUID PRIMARY KEY,
  source      TEXT NOT NULL,           -- 'chatgpt' | 'claude' | 'gemini'
  external_id TEXT NOT NULL,           -- id с того сервиса
  title       TEXT,
  started_at  TIMESTAMPTZ,
  updated_at  TIMESTAMPTZ DEFAULT NOW(),
  raw_json    JSONB,
  UNIQUE(source, external_id)
);
CREATE TABLE messages (
  id              UUID PRIMARY KEY,
  conversation_id UUID REFERENCES conversations(id) ON DELETE CASCADE,
  role            TEXT,                -- 'user' | 'assistant' | 'system'
  content         TEXT,
  position        INT,
  sent_at         TIMESTAMPTZ,
  content_tsv     TSVECTOR
);
CREATE INDEX ON messages USING GIN(content_tsv);

4

Перехватчики для трёх AI-сайтов

~2-3 дня (самая нетривиальная часть)

Главный подход — перехватывать fetch-ответы через chrome.webRequest.onCompleted или через injected script, патчащий window.fetch. Это надёжнее парсинга DOM.

  • Claude.ai: следить за GET /api/organizations/{org}/chat_conversations/{convo} — там полный разговор в JSON
  • ChatGPT: GET /backend-api/conversation/{id}
  • Gemini: самый сложный, ответы идут стримом, нужно собирать по chunks
  • Для каждого источника — отдельный нормализатор в backend, приводящий JSON к единому формату
// tip

Не парси DOM. Любое изменение фронтенда сайта сломает твой экстеншн. Перехват fetch стабильнее на порядок.

~1-2 дня

  • Страница /c/[id] — рендер conversation, markdown поддержка
  • Search bar сверху — текстовый поиск через GET /search
  • Подсветка совпадений в результатах

6

Дедупликация и стабилизация

~½ дня

UPSERT по (source, external_id) — один и тот же разговор может приходить много раз при перезагрузке страниц. Также — обработка обновлений (новые сообщения в существующем чате).

// pitfalls
  • Manifest V3 запретил persistent background scripts — используй service workers и offscreen documents для долгих задач
  • CORS: extension не может ходить в localhost:8000 без явного указания в manifest host_permissions
  • ChatGPT требует Authorization Bearer токен — он есть в куках, но обновляется; обновляй парсер каждые пару недель
  • Gemini обновляется чаще всех — самый хрупкий источник
// acceptance criteria

Открываешь chatgpt.com, ведёшь короткий разговор, закрываешь вкладку. Через 5 секунд в web-UI на localhost:3000 появляется этот разговор, его можно открыть, видны все сообщения, поиск находит его по словам из переписки.

  • Web-UI на localhost:3000 для просмотра и текстового поиска
  • Никаких внешних API, нулевые траты
PHASE 02

RAG слой — семантический поиск

~1-2 недели

Поиск становится умным. «Найди где я разбирал auth в Next.js» работает, даже если в разговорах не было слова "auth" — было "аутентификация" или "logging in".

// Что нового

Ollama (локально) nomic-embed-text pgvector embeddings Chunking pipeline

// Шаги

1

Установка Ollama

~½ дня
  • Download с ollama.com
  • ollama pull nomic-embed-text (274 MB, идёт на 8GB RAM)
  • Альтернатива для лучшего качества на русском: ollama pull bge-m3 (хотя 1.2GB)
  • Проверка: curl http://localhost:11434/api/embeddings -d '{"model": "nomic-embed-text", "prompt": "test"}'
  • Ollama-клиент для Python: pip install ollama
2

Chunking strategy

~1 день

Длинные сообщения нужно резать. Стратегия:

  • Каждое сообщение — отдельный chunk (часто этого достаточно)
  • Если сообщение > 2000 символов — режем на 1500-символьные куски с 200-символьным overlap
  • Отдельная таблица chunks: id, message_id, position, content, embedding vector(768)
  • Никогда не теряем связь chunk → message → conversation
CREATE TABLE chunks (
  id          UUID PRIMARY KEY,
  message_id  UUID REFERENCES messages(id) ON DELETE CASCADE,
  position    INT,
  content     TEXT NOT NULL,
  embedding   vector(768),   -- размерность nomic-embed-text
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
-- не индексируем embedding пока кусков мало (< 50k)
3

Embedding pipeline

~1 день

Background worker, который берёт неэмбедженные chunks и обрабатывает их.

  • После UPSERT нового сообщения — задача в очередь (arq или просто asyncio task)
  • Worker берёт чанки без embedding, шлёт в Ollama, сохраняет
  • Batch размером 32 для производительности
  • Также — задача «переэмбедь всё» для случая когда меняешь модель
4

Vector search endpoint

~1 день

Новый endpoint POST /search/semantic:

  • Принимает query текстом
  • Эмбедит query через Ollama
  • SQL: SELECT * FROM chunks ORDER BY embedding <-> $1 LIMIT 20
  • Возвращает chunks с контекстом (id message, conversation, дата, source)
// tip — hybrid search

Лучшие результаты даёт гибрид текстового и семантического поиска. Считаешь оба, ранжируешь через RRF (Reciprocal Rank Fusion). Реализуется в 10 строк SQL и заметно поднимает качество.

5

Frontend update

~1 день
  • Search box с переключателем режимов: текст / семантика / гибрид
  • Карточки результатов с подсветкой найденного фрагмента
  • Кнопка «найти похожие» на каждом сообщении (через embedding текущего message)
// pitfalls
  • Размер chunks: слишком маленькие — теряется контекст; слишком большие — ухудшается поиск. Начни с message-as-chunk и уменьшай при необходимости.
  • pgvector index: для маленьких БД (до ~50k chunks) не индексируй — full scan быстрее. Индекс HNSW тратит память.
  • Языки: nomic-embed-text неплохо понимает русский, но если разговоры в основном на нём — попробуй bge-m3 или multilingual-e5
  • Переэмбеддинг при смене модели: предусмотри сразу, иначе будешь думать «почему всё сломалось»
// acceptance criteria

В поиске вводишь «как защитить эндпоинт от чужих запросов» — находит твой разговор, где ты обсуждал middleware и JWT, хотя слов из запроса там не было.

// что появилось после Phase 2
  • Локальная Ollama с embedding-моделью
  • Pipeline эмбеддинга всей твоей истории
  • Семантический поиск по смыслу, не по словам
  • Никаких новых внешних API, всё ещё нулевые траты
PHASE 03

Memory Layer — извлечение фактов

~2-3 недели

AI начинает извлекать структурированные факты о тебе из разговоров. Через 2-3 недели использования у тебя есть профиль с десятками фактов — что ты знаешь, чем интересуешься, над чем работаешь, с кем общаешься.

// Что нового

Gemini API (free tier) Extraction worker profile_facts таблица Profile UI

// Шаги

1

Gemini API setup

~½ дня
  • Аккаунт на Google AI Studio (бесплатно, без карты)
  • API ключ → в .env файл
  • pip install google-genai
  • Минимальная обёртка с retry, rate limiting (15 RPM для Flash-Lite)
  • Лучшая модель для extraction: gemini-2.5-flash (10 RPM, 250 RPD бесплатно) или gemini-flash-lite (15 RPM, 1500 RPD)
2

Extraction prompt

~2-3 дня (итеративно)

Самая нетривиальная часть. Промпт нужно итерировать на реальных данных. Output — strict JSON schema.

# Output schema
{
  "skills": [{name, level, source_message_id}],
  "preferences": [{topic, opinion, source_message_id}],
  "projects": [{name, description, status, source_message_id}],
  "people": [{name, relation, source_message_id}],
  "facts": [{statement, category, source_message_id}]
}
  • System prompt пишется на английском (модели лучше его понимают), контент в нём может быть на русском
  • Использовать Gemini structured output (JSON mode) для надёжности
  • Тестировать на 20-30 реальных разговорах, итерировать промпт
3

Schema для фактов

~1 день
CREATE TABLE profile_facts (
  id                UUID PRIMARY KEY,
  category          TEXT,           -- skills, preferences, projects, people, facts
  content           TEXT NOT NULL,
  source_message_id UUID REFERENCES messages(id),
  extracted_at      TIMESTAMPTZ DEFAULT NOW(),
  confidence        REAL,           -- 0..1
  manually_edited   BOOLEAN DEFAULT FALSE,
  deleted           BOOLEAN DEFAULT FALSE,
  embedding         vector(768)     -- для семантического поиска фактов
);
CREATE INDEX ON profile_facts(category) WHERE NOT deleted;
4

Extraction worker

~1-2 дня
  • При appearance нового разговора → задача в очередь
  • Worker берёт разговор, формирует промпт, шлёт в Gemini
  • Парсит JSON, сохраняет факты в profile_facts
  • Дедупликация: перед вставкой ищет похожий факт через embedding similarity (> 0.85 = дубль, обновляем существующий)
  • Очередь работает в фоне, без блокировок основного flow
5

Conflict resolution

~1-2 дня

Что если в марте «я люблю TypeScript», а в августе «я устал от TypeScript»? Логика:

  • Не удаляем старый факт — храним оба с метками времени
  • UI показывает «текущее мнение» (последнее по дате)
  • В hover/expand видна история: «было — стало»
  • Это исторический архив тебя, а не текущее состояние
6

Profile UI

~1 день
  • Страница /me с категориями (Skills / Preferences / Projects / People / Facts)
  • В каждой категории — список фактов
  • На каждом факте: edit, delete, «не верно» (помечает как deleted, но AI больше не извлекает аналогичный)
  • Клик на факт → переход к оригинальному сообщению (где он был извлечён)
  • Сверху — статистика «N фактов о вас, последний добавлен X дней назад»
// pitfalls
  • Hallucination: модель может извлечь факт, которого не было. Поэтому всегда нужен source_message_id для проверки
  • Prompt injection: твои разговоры могут содержать «ignore previous instructions» — оборачивай user-content в чёткие разделители, не доверяй вложенным инструкциям
  • Rate limits Gemini: 15 RPM = 1 запрос в 4 секунды. Реализуй очередь с задержками
  • Russian language: качество extraction на русском хуже. Можно перевести контент на английский перед extraction, но это лишний API call — пробуй сначала на русском
  • Privacy для уязвимых фактов: добавь категорию «sensitive» и не показывай её в общем UI без специального тумблера
// acceptance criteria

Использовал систему 2 недели → открыл /me → видишь 30-50+ корректных фактов о себе, разнесённых по 5 категориям. На каждом факте кликаешь — попадаешь на оригинал в разговоре.

// что появилось после Phase 3
  • Локальный сервис с твоим профилем (структурированная база фактов о тебе)
  • Автоматическое наращивание профиля без твоего участия
  • UI редактирования профиля
  • Внешний API: Gemini Free Tier ($0/мес при умеренном использовании)
PHASE 04

Chat with your AI — финальный продукт

~1-2 недели

Чат-интерфейс, в котором AI знает твою историю и факты о тебе. На любой вопрос он подтягивает релевантный контекст и отвечает с цитатами из реальных твоих разговоров.

// Что нового

Chat endpoint (SSE streaming) RAG retrieval pipeline Claude / Groq для chat Chat UI с подсветкой источников

// Шаги

1

Chat backend endpoint

~2-3 дня

POST /chat со streaming-ответом через Server-Sent Events. На каждый запрос pipeline:

  1. Эмбедим query пользователя через Ollama
  2. Top-K (10-15) релевантных chunks из истории через pgvector
  3. Top-K (5-10) релевантных facts из профиля через embedding similarity
  4. Конструируем system prompt с этим контекстом
  5. Шлём в LLM, стримим обратно
  6. Возвращаем вместе с ответом — список использованных источников (chunk_ids)
2

Выбор LLM

~½ дня

Стратегия от бесплатного к платному:

  • Free start: Groq с Llama 3.3 70B — быстро, бесплатно, лимиты щадящие
  • Cheap: Claude Haiku ($0.25/M input) — качество ощутимо выше, копейки
  • Production: Claude Sonnet ($3/M) — если станет основным инструментом
  • Конфиг переключается одной переменной в .env
  • Anthropic prompt caching для system prompt — резко снижает стоимость при многократных запросах
3

Conversation memory внутри чата

~1 день
  • Текущий разговор с твоим AI хранится в отдельной таблице pam_sessions
  • Не путать с conversations (это твоя история с другими AI)
  • В каждом запросе передаём последние N сообщений из текущей сессии
  • Длинные сессии — checkpoint и суммаризация для экономии токенов
4

Chat UI

~2-3 дня
  • Страница /chat в Next.js
  • Streaming markdown через react-markdown + Server-Sent Events
  • Под каждым ответом — chips «использовано: разговор с Claude от 12 марта, факт из профиля»
  • Кнопка «почему так?» открывает sidebar с подробностями retrieval'a
  • Возможность копировать, заиметь shareable link на сессию
5

Refinement и tuning

~2-3 дня
  • Промпт-инжиниринг для финального chat'a
  • Тестирование на real questions — где retrieval хороший, где плохой
  • Logging каждого retrieval'a для дебага
  • Контроль temperature, max_tokens
  • Edge cases: «что если ничего не нашлось» — честно говорим, не выдумываем
// pitfalls
  • Context window: даже у Sonnet 200k, но это не значит что туда нужно пихать всё. Лучше 10 хорошо ранжированных chunks, чем 100 случайных
  • Retrieval quality: если retrieval плохой — AI отвечает мусором. Логируй каждый запрос и руками проверяй на проблемных кейсах
  • Cost при активном использовании: chat — самая дорогая операция. Без prompt caching можешь спалить $20-50/мес
  • Latency: streaming решает UX, но первый токен всё равно важен. Кэшируй embeddings query, переиспользуй открытые соединения
  • Hallucination в финальном ответе: если AI выдумал «вы говорили X» — нужна явная проверка по source_ids в pipeline
// acceptance criteria

Спросил «что ты помнишь про мой проект Y» — AI пересказывает с цитатами из старых разговоров на эту тему. Спросил «какие у меня были основные проблемы с auth в проектах в этом году» — AI агрегирует фрагменты из нескольких разных разговоров.

// что появилось после Phase 4 — финальный продукт
  • Полнофункциональный AI-ассистент, который реально тебя знает
  • Local-first архитектура, твои данные у тебя
  • Расходы $0-15/мес при активном использовании
  • Это инструмент, который растёт со временем — чем дольше используешь, тем умнее
// risks

Главные риски и митигации

// Технические

// extension хрупкий

Что случится: рано или поздно один из AI-сайтов поменяет своё API. Твой перехватчик сломается на 1-3 дня, пока ты не починишь.

Митигация: 1) логи всего извлечения с алертом «N часов нет новых разговоров»; 2) три источника изолированы — отвалится один, остальные работают; 3) добавь возможность manual import через JSON-экспорт ChatGPT/Claude (они отдают export по запросу).

// extraction quality

Что случится: на ранних разговорах AI извлечёт много ерунды или повторяющихся фактов.

Митигация: 1) первую неделю активно чистишь профиль вручную, AI учится на твоих edit'ах через few-shot prompt; 2) confidence-фильтр (не показывай факты ниже 0.6); 3) regular re-extraction по обновлённому промпту.

Что случится: в Phase 3 и 4 ты неминуемо будешь менять схему. Если делать без Alembic — потеряешь данные.

Митигация: с самого Phase 1 ставь Alembic, даже если кажется overkill. Каждое изменение схемы — миграция. Бэкап БД (просто pg_dump в файл) делай еженедельно.

// vendor lock-in на LLM

Что случится: завязался на Gemini Flash, потом он подорожал в 5 раз / у тебя забанили / Google убрал бесплатный tier.

Митигация: с самого начала — абстракция LLMClient с провайдерами. Переключение через ENV. Тестируй pipeline на 2 провайдерах сразу.

// Психологические

// уязвимый профиль

Что случится: через месяц у AI набирается такой профиль тебя, что становится неприятно видеть это в одном месте. «AI знает что я ленивый», «AI помнит, что я с этим человеком в ссоре».

Митигация: возможность пометить категорию как «sensitive» и спрятать. Возможность массово удалить факты по фильтру. Эта часть продукта про эмоциональный комфорт, не только про функциональность.

// scope creep

Что случится: на Phase 3 ты захочешь подключить ещё email, Notion, calendar — и Phase 4 не наступит никогда.

Митигация: жёстко идти по плану. Сначала закрыть все 4 фазы только с AI-чатами как источником. Подключение других источников — Phase 5, после полностью рабочего MVP.

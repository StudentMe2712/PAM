# Claude Code — практики для работы над PAM

Выжимка применимого из репо-справочника
[shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice)
(там же — полный список из 83 советов, сравнение воркфлоу, cross-model и т.д.).
Здесь — только то, что реально используем в PAM. Демо-ассеты того репо
(weather/time/presentation агенты, звуковые хуки, чужие settings) НЕ копировали.

## Рабочий цикл (мы уже так и делаем — GSD)
**Research → Plan → Execute → Review → Ship.** В PAM это: `.planning/ROADMAP.md`
(план фаз) → ветка на фазу → мелкие atomic-коммиты → `/code-review`+`/security-review`
→ мерж в `main`. Состояние — в `.planning/STATE.md`, журнал — `handoff.md`.

## Планирование
- Начинать со **spec/плана**, снижать неоднозначность до старта.
- **Вертикальные срезы** (DB + сервис + UI разом), а не горизонтальные фазы —
  чтобы end-to-end фидбек был раньше. *(Учесть: мы местами шли горизонтально.)*
- Фазы с **гейтами и тестами** на каждой (у нас так и есть).
- Второй Claude/Codex как «staff engineer» на ревью плана.

## Контекст (важно для длинных сессий — у нас они длинные)
- «Тупая зона» ~40% контекста: новичкам держать <40%, опытным <30%; на простых
  задачах можно до 60%. Переключаешь задачу — `/clear` или `/compact`.
- **Новая задача = новая сессия.** Родственную (доки по тому, что сделал) — можно в той же.
- `/compact` с подсказкой («focus on …, drop …») лучше авто-компакта.
- **Субагенты для управления контекстом**: спроси «нужен ли потом сам вывод тула
  или только вывод?» — 20 чтений файлов + грепы остаются в контексте субагента,
  наружу возвращается только итог. (У нас: Explore/Plan/general-purpose.)
- `/rewind` вместо «правлю поверх ошибки» — не засоряй контекст неудачными попытками.

## CLAUDE.md / память
- CLAUDE.md — целиться в **<200 строк** на файл (короче = надёжнее исполняется).
- Большие инструкции — выносить в `.claude/rules/*.md` (можно с `paths:` для ленивой загрузки по glob).
- Любой разработчик должен сказать «run the tests» и оно сработает с первого раза —
  иначе в CLAUDE.md не хватает команд setup/build/test. *(У нас команды есть в CLAUDE.md.)*
- Harness-поведение (attribution, permissions, model) — через `settings.json`, а не текстом в CLAUDE.md.
- Авто-память: `~/.claude/projects/<project>/memory/` (мы её ведём — `MEMORY.md` + файлы).

## Промптинг / отладка
- «Grill me on these changes / prove this works» — пусть Claude докажет, что работает (diff main↔branch).
- После посредственного фикса: «зная всё, что теперь знаешь, выкинь это и сделай элегантно».
- Большинство багов: вставь баг → «fix», не микроменеджь как именно.

## Git / сессии
- Значимые изменения → отдельная ветка с понятным именем + понятные коммиты (наше правило).
- «summarize from here» перед `/rewind` — Claude пишет хэндофф-заметку себе.
- Длинные сессии — recaps (короткие сводки «что сделал / что дальше»).

## Не тащим в PAM (намеренно)
Звуковые хуки (.mp3 на каждое событие), демо-агенты/скиллы (weather/time/presentation),
чужой `settings.json` (кастомный спиннер, чужие permissions), презентации, changelog,
Codex-конфиги. Это материал для обучения, а не ассеты нашего проекта.
